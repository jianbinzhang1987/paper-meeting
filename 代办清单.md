# 无纸化会议系统客户端代办清单

## 说明

- 当前客户端主工程为 `paper-meeting-client`，技术栈为 Flutter。
- 当前已接入的客户端接口只有：
  - `GET /meeting/app/bootstrap`
  - `POST /meeting/app/sign-in`
  - `GET /meeting/app/documents`
  - `GET /meeting/app/notices`
  - `GET /meeting/app/votes`
  - `POST /meeting/app/vote/submit`
- 当前大量功能处于 `UI 已做 / 本地状态可演示 / 未形成完整业务闭环` 状态。

## P0：优先补齐，决定客户端是否可真正上线

- [x] 文稿阅读器接入真实 PDF 引擎。
- [x] 文稿列表接口补充页数、文件类型、文件 URL、缩略图等完整字段。
- [x] 文稿页支持真实翻页、缩放、跳页，而不是仅靠本地页码变量模拟。
- [x] 文稿批注、书签实现本地持久化，至少重进应用不丢失。
- [ ] 批注和书签如果需要跨端同步，补充后端保存与查询接口。
- [x] 签名提交改为真实上传，不再使用“发一条通知”代替提交。
- [x] 后端补充签名上传、签名记录查询接口。
- [x] 前端签名支持提交成功、失败、重复提交、清空重签等状态提示。
- [ ] 视频模块接入真实播放能力，支持视频在线播放或受控播放。
- [x] 资料库模块接入真实下载/打开/预览能力，移除当前禁用按钮。
- [x] 设备配置、签到信息、最近一次会议基础信息做本地持久化。
- [x] 断网重连改为真实网络状态检测，不再只做按钮模拟。
- [x] 资料预加载改为真实下载缓存，不再只做进度条动画。

## P1：核心业务闭环，当前大多只有演示态

- [x] 秘书端“开始表决”调用后端真实接口，而不是只改本地状态。
- [x] 秘书端“结束表决”调用后端真实接口。
- [x] 秘书端“发布结果”调用后端真实接口。
- [x] 客户端表决结果刷新时展示真实票数和当前用户是否已投状态。
- [ ] 若支持匿名表决，前后端统一匿名逻辑和结果展示规则。
- [x] 广播消息改为真实发布接口或 WebSocket 下发，不再只插入本地通知列表。
- [x] 强制返回同屏改为真实服务端广播控制。
- [x] 强制停止同屏改为真实服务端广播控制。
- [x] 强制注销签到补充完整协议、服务端消息类型和客户端处理逻辑。
- [x] 客户端补齐 `FORCE_LOGOUT` 处理，收到消息后应清理会话并回到签到页。
- [x] 同屏申请、审批、拒绝、停止统一消息结构，避免前后端字段不一致。
- [x] 同屏状态要能区分“发起人”“审批人”“跟随者”“自由阅读者”。
- [ ] 服务呼叫请求补充历史记录查询和状态流转记录。
- [ ] 服务处理结果需要明确已接单、处理中、已完成、已取消等状态定义。
- [ ] 会议通知支持已读状态、未读提示和多条通知稳定分页。

## P2：可用性和体验增强

- [x] 会议信息页区分应到、已签到、在线人数，不能都直接使用用户总数。
- [x] 登录页支持按座位自动筛选账号，减少平板误选。
- [ ] 账号签到时区分会议密码和个人密码逻辑，避免后续扩展冲突。
- [x] 文稿区的动态水印改为真实随页渲染，而不是简单覆盖文字。
- [ ] 视频同屏支持播放、暂停、进度同步，而不是单次“同步演示”动作。
- [x] 资料库支持分类、搜索、最近访问、已下载标记。
- [x] 通知页支持自动轮播开关和置顶通知。
- [ ] 计时器如用于会议发言控制，建议补充秘书端下发统一计时。
- [x] 实时消息日志增加筛选、级别和导出能力，便于排查联调问题。
- [ ] 增加错误态页面：无会议、无文稿、无通知、接口失败、WebSocket 断开。
- [x] 已补基础空态与错误提示：无文稿、无通知、无表决、无资料、无服务回执、接口失败提示。

## 后端接口补充建议

- [x] 新增签名上传接口，例如 `/meeting/app/signature/submit`。
- [ ] 新增资料下载或授权访问接口，明确文件访问鉴权。
- [x] 新增广播消息发布接口，或明确由管理端发布后通过 WebSocket 下发客户端。
- [ ] 新增客户端控制接口集合，统一处理同屏、强制返回、强制退出、视频控制。
- [ ] 若秘书控制只走管理端接口，客户端需要补充鉴权和角色校验方案。
- [x] 为 `bootstrap` 返回的文稿、视频、资料补齐展示所需字段，避免前端二次猜测。
- [x] 为 WebSocket 定义固定协议文档，至少包含消息类型、字段说明、示例载荷。
- [x] 为客户端提供“会议实时状态”查询接口，便于断线重连后恢复现场。

## 前端代码层面的直接问题

- [x] `DocumentPanel` 当前主阅读区仍是占位文案，需要替换为真实阅读组件。
- [x] `ApiMeetingRepository` 当前将文稿页数写死为 `1`，需要改为服务端真实字段。
- [x] `ApiMeetingRepository` 当前只把文稿 URL 拼进 `summary`，需要单独模型字段承载。
- [x] `submitSignature()` 当前只是调用 `broadcastNotice()`，需要改为真实业务提交。
- [x] `broadcastNotice()` 当前只改本地 `session.notices`，未真正发到服务端。
- [x] `forceReturnSync()` 当前只改本地状态，未发送控制消息。
- [x] `startVote()`、`finishVote()`、`publishVote()` 当前只更新本地状态。
- [x] `toggleNetworkStatus()` 当前是模拟开关，不是真实网络探测。
- [x] `simulatePreload()` 当前是模拟进度，不是真实缓存。
- [x] 客户端当前没有本地存储方案，配置和用户操作记录会丢失。

## 联调顺序建议

- [ ] 第一阶段：先打通 `文稿真实预览 + 签名上传 + 视频播放 + 资料下载`。
- [ ] 第二阶段：打通 `秘书端表决控制 + 广播通知 + 强制同屏控制`。
- [ ] 第三阶段：补 `断点恢复 + 本地缓存 + 异常页 + 持久化`。
- [ ] 第四阶段：补 `自动化测试 + 真机联调 + 发布流程`。

## 验收标准建议

- [x] 设备首次配置后，关闭应用再次进入仍可恢复服务器和座位配置。
- [x] 用户签到成功后，刷新或重进应用可恢复当前会议上下文。
- [x] 文稿可以真实打开 PDF，支持翻页、跳页、缩放。
- [x] 签名提交后服务端可查询到签名结果。
- [ ] 视频可播放，资料可下载或预览。
- [ ] 秘书发起表决后，参会端能实时收到状态变化。
- [ ] 秘书发布通知后，所有客户端能实时接收。
- [ ] 强制返回同屏、强制停止同屏、强制注销能在所有终端生效。
- [x] 断网重连后，客户端能够恢复会议通知、服务请求、同屏状态等实时信息。
- [ ] 至少补齐基础 widget test、控制器单测和一次真机联调清单。
