# 04 客户端联调与信令协议设计

## 1. 目标

本文档用于约束 `paper-meeting-client` 与 `paper-meeting-server` 的客户端联调边界，重点覆盖：

1. 客户端启动与签到所需 REST 接口
2. 会中实时控制所需 WebSocket 消息协议
3. 断网重连、资料预加载与状态恢复策略

---

## 2. 当前工程现状

### 2.1 已存在后台接口

管理端侧已具备以下会议域接口：

- `/meeting/info/*`
- `/meeting/agenda/*`
- `/meeting/attendee/*`
- `/meeting/file/*`
- `/meeting/vote/*`
- `/meeting/notification/*`
- `/meeting/room/*`

上述接口主要面向后台管理端使用，后续客户端建议新增 `/meeting/app/*` 轻量接口，避免直接暴露后台管理模型。

### 2.2 已落地客户端接口

当前版本已经完成以下接口落地：

- `GET /meeting/app/bootstrap`
- `POST /meeting/app/sign-in`
- `GET /meeting/app/documents`
- `GET /meeting/app/notices`
- `GET /meeting/app/votes`
- `POST /meeting/app/vote/submit`
- `POST /meeting/vote/start`
- `POST /meeting/vote/finish`
- `POST /meeting/vote/publish`
- `POST /meeting/vote/force-return`

### 2.3 当前客户端工程

`paper-meeting-client` 当前已经实现：

- 初始化配置
- 登录签到
- 会议信息/文稿/视频/资料库
- 同屏申请与跟随
- 表决、签名、呼叫服务、通知、计时器
- 秘书控制台
- 断网状态切换模拟
- 资料预加载进度模拟
- 登录后基于签到返回 token 自动连接 `/infra/ws`
- 通知发布、表决状态、强制返回的实时消息接收与日志落盘

## 3. 建议 REST 接口

### 3.1 初始化与登录

#### `GET /meeting/app/bootstrap`

用途：根据设备配置的会议室、座位、设备号返回当前终端可进入的会议信息。

建议参数：

- `roomName`
- `seatName`
- `deviceName`

建议返回：

```json
{
  "meetingId": 1001,
  "meetingName": "集团办公会（2026年第5次）",
  "description": "季度经营与预算审议会议",
  "controlType": 1,
  "watermark": true,
  "topics": [],
  "documents": [],
  "users": [],
  "votes": [],
  "notices": []
}
```

#### `POST /meeting/app/sign-in`

用途：客户端账号选择后完成签到并返回当前用户角色。

建议参数：

```json
{
  "meetingId": 1001,
  "userId": 2001,
  "password": "123456",
  "deviceName": "华为平板-01"
}
```

当前实际返回已补充：

- `accessToken`
- `accessTokenExpiresTime`
- `websocketPath`

### 3.2 文稿与资料

#### `GET /meeting/app/document/list`

用途：获取当前会议所有议题文稿、视频和公共资料列表。

#### `POST /meeting/app/document/bookmark`

用途：保存书签。

#### `POST /meeting/app/document/note`

用途：保存文本注释。

#### `POST /meeting/app/document/preload`

用途：上报客户端资料预加载结果与本地版本。

### 3.3 表决与签名

#### `POST /meeting/app/vote/submit`

用途：参会者提交表决。

#### `POST /meeting/app/signature/submit`

用途：上传电子签名图片或矢量轨迹。

### 3.4 呼叫服务

#### `POST /meeting/app/service/request`

用途：提交会场服务请求。

#### `GET /meeting/app/service/list`

用途：查询当前座位的服务请求及处理状态。

## 4. WebSocket 消息协议

建议使用统一消息包：

```json
{
  "type": "SYNC_REQUEST",
  "meetingId": "1001",
  "timestamp": "2026-03-26T10:30:00",
  "payload": {}
}
```

当前实现中客户端实际订阅地址为：

- `ws://{host}:{port}/infra/ws?token={accessToken}`
- 当端口为 `443` 时自动切换为 `wss://`

### 4.1 消息类型

| 类型 | 说明 |
| --- | --- |
| `SYNC_REQUEST` | 普通参会者发起同屏申请 |
| `SYNC_APPROVED` | 秘书审批通过，开始同屏 |
| `SYNC_STOPPED` | 停止同屏 |
| `VOTE_STARTED` | 开始表决 |
| `VOTE_FINISHED` | 结束表决 |
| `VOTE_PUBLISHED` | 发布表决结果 |
| `NOTICE_PUBLISHED` | 发布会议通知 |
| `SERVICE_UPDATED` | 呼叫服务状态回执 |
| `FORCE_RETURN` | 强制返回同屏 |
| `FORCE_LOGOUT` | 强制注销签到 |

### 4.2 典型 payload

#### `SYNC_REQUEST`

```json
{
  "requestId": "sr-10001",
  "userId": 2001,
  "userName": "王强",
  "documentId": "d-101",
  "documentTitle": "重点项目进度简报",
  "page": 5
}
```

#### `SYNC_APPROVED`

```json
{
  "requestId": "sr-10001",
  "leaderUserId": 2001,
  "documentId": "d-101",
  "page": 5,
  "zoom": 1.0
}
```

#### `NOTICE_PUBLISHED`

```json
{
  "noticeId": 301,
  "title": "会议广播",
  "content": "请相关人员返回同屏，准备进入下一项议题。"
}
```

## 6. 断网重连与恢复

1. 断网后立即进入 `offline` 状态并提示用户。
2. 自动触发指数退避重连，状态变更为 `reconnecting`。
3. 重连成功后重新拉取：
   - 当前会议主状态
   - 当前同屏主讲页码
   - 当前表决状态
   - 最新通知
4. 若处于强制同屏状态，客户端自动跳转回主讲页。
