# 详细设计说明书 - Phase 2: 文件处理与实时信令

## 1. 文件处理流水线设计

### 1.1 自动化转换流程 (File Pipeline)
1.  **上传处理**：文件进入 `ruoyi-modules-file` 模块，保存至本地存储路径（基于 Nginx 静态目录映射）。
2.  **异步转换 (Spring @Async)**：
    -   调用 **LibreOffice (Headless)** 将 `docx/pptx` 转换为 `PDF`。
    -   进一步将 `PDF` 每页切分为 `PNG/JPEG` 高清图（支持缩放适配）。
3.  **加密封装 (dtq)**：
    -   将切分后的图片、附件及元数据（MD5、页码）进行物理分片并采用 AES-256 加密。
    -   封包文件后缀为 `.dtq`。仅允许内建渲染器解密，文件拉取时强制绑定 Token。
4.  **水印叠加**：根据会议配置，在下载加密包时动态生成当前 `user_id` + `time` 的全屏斜向水印（Overlay）。

## 2. 实时信令同步 (Sync Engine)

### 2.1 WebSocket 架构
1.  **连接管理**：采用 Spring Boot WebSocket 或 Netty-WebSocket。连接地址：`ws://server_ip:9394/meeting/sync/{meetingId}`。
2.  **指令下发流程 (Mode A: 指令级)**：
    -   **主讲人端**：发送 JSON 报文：`{"op": "PAGING", "fileId": 123, "pageIdx": 5, "scaleX": 1.0, "scaleY": 1.0}`。
    -   **服务端**：在 Redis 缓存中更新该会议的 `sync_state`（Key: `mt_sync:{meetingId}`），并向全场 Session 广播该报文。
    -   **跟随端**：解析报文，本地渲染引擎跳转至对应页面。
3.  **手写同步**：
    -   轨迹数据报文：`{"op": "INK", "points": [x1,y1, p1, x2,y2, p2, ...], "color": "#FF0000"}`。
    -   采用频率限制（如 20ms 一组）合并点位。

## 4. 专项优化方案
- **断网恢复**：客户端重连后，主动从 `Redis (Key: mt_sync:{meetingId})` 请求最新操作报文。
- **离线包预加载**：会前 10 分钟或秘书操作“发放资料”时，客户端开始 Service 静默下载 `.dtq` 包，优先当前议程。
